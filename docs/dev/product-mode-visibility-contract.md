# Product Mode Visibility Contract

Normal VoidEmpires UI defaults to product mode. Product mode is the experience shown by every route unless the local browser session explicitly reveals operator/internal controls.

This contract is documentation-only. It does not remove internal endpoints, scripts, or Development tools needed for local operation.

## Modes

| Mode | Default | Audience | Allowed visible content |
|---|---|---|---|
| Product | Yes | Normal local player/demo user | Player-facing route navigation, colony/map/readiness state, guarded gameplay confirmations, non-technical limitations. |
| Operator/Internal | No | Developer or QA operator in a local browser session | Diagnostics, raw IDs, backend route/profile details, JSON payload previews, action manifests, materialization controls, and Development-only mutation helpers. |

Product mode is the only mode allowed for first render, direct route visits, copied links, and normal sidebar navigation.

## Product Mode Rules

- Do not show `Development`, `QA`, `backend`, `endpoint`, `payload`, `raw`, `localhost`, `provider`, `seed`, `database`, `base de datos`, `mock`, `placeholder`, `prototipo`, or `mutacion` in primary UI.
- Do not show raw GUIDs, endpoint paths, backend profiles, JSON payloads, or stack/request details in first-viewport content or primary cards.
- Do not render `DevEndpointNotice`, `DevelopmentToolsPanel`, `DevDiagnosticsPanel`, or `ActionManifestPanel` in product mode.
- Do not show materialization controls, action manifests, direct transfer tooling, or technical complete-due controls in product mode.
- Do not persist or display auth-like state. Local playable-session memory may rebuild navigation context, but it is not login, ownership proof, a role, a token, or production authorization.
- Do not overclaim final systems: production authentication, public ranking, market transactions, alliance mutations, combat, invasion, interception, and final fleet movement productization stay out of product-mode claims.

## Operator/Internal Reveal Contract

Allowed reveal mechanisms must be local, explicit, and non-sensitive:

- Query flag for one navigation session: `?operator=1`.
- Local browser flag for repeated local work: `localStorage["voidempires.operatorMode"] = "1"`.
- A future in-app toggle may set the same local flag only after an explicit user action.

Required behavior for any implementation:

- Product mode is active when no allowed reveal mechanism is present.
- Operator mode must never be inferred from API responses, user IDs, player names, email, auth state, remote configuration, or backend profile text.
- The local storage flag must store only the string `"1"` or be absent. It must not store credentials, roles, claims, tokens, emails, endpoint URLs, or raw payloads.
- Query flag reveal is URL-local. Product links generated by the app should not preserve `operator=1` unless the link opens an operator-only panel or tool.
- Operator mode can reveal technical UI, but it must not auto-run mutations, materialization, queue completion, transfers, resource changes, or other state-changing calls.
- Operator controls still require the same explicit confirmation surfaces used today.

## Components Covered

| Component/surface | Product mode | Operator/Internal mode |
|---|---|---|
| `AppShell` status copy | Player-facing context only. No backend profile, Development mode, or production-login caveat. | May show environment/profile details. |
| `DevEndpointNotice` | Hidden. | May show API base URL and expected backend profile. |
| `DevelopmentToolsPanel` | Hidden. | May show materialization and local operator actions, closed unless explicitly opened. |
| `DevDiagnosticsPanel` | Hidden. | May show summaries, raw IDs, and JSON payload preview. |
| `ActionManifestPanel` | Hidden. | May show contract metadata and read-only/mutating action availability. |
| Route ID forms | Product labels and local context recovery only. | May expose exact `civilizationId`, `planetId`, `systemId`, tactical IDs, and transfer IDs. |
| Error details | Short player-facing recovery text. | May expose technical message and raw response detail. |

## Route Requirements

- `/onboarding`: product mode presents a local start flow without `Development-only` or backend details. Operator mode may expose returned IDs and request failures.
- `/` and `/galaxy`: product mode shows the strategic map and selected context. Operator mode may expose payloads, visual seed, and system/planet diagnostics.
- `/planet` and `/construction`: product mode shows colony state, construction queue, and guarded order confirmation. Operator mode may expose resource/materialization tools and diagnostics.
- `/research`: product mode shows catalog, queue, and guarded research confirmation. Operator mode may expose raw diagnostics and backend acceptance details.
- `/shipyard`: product mode shows orbital production and queue readiness. Operator mode may expose materialization guidance and raw queue/stock payloads.
- `/fleets`: product mode shows fleet/readiness summaries. Operator mode may expose transfer tooling, action manifests, tactical IDs, and raw transfer diagnostics.
- `/defenses` and `/ground-army`: product mode shows readiness and scope boundaries. Operator mode may expose technical IDs and route limitations.
- `/market`, `/espionage`, `/alliance`, `/ranking`: product mode stays advisory/read-only. Operator mode may expose context IDs, comparison diagnostics, and raw limitations.

## Implementation Guidance For Follow-Ups

- Prefer one shared helper, for example `isOperatorMode(searchParams)`, instead of route-specific checks.
- Keep storage access centralized, following the existing `playableSession.ts` pattern.
- Preserve route lazy loading from `App.tsx`.
- Treat operator mode as a visibility layer only. It is not an authorization layer and must not weaken existing confirmation or backend validation.
- Update `scripts/check-frontend-copy-regressions.ps1` when product-mode hiding is implemented so forbidden technical copy is allowed only in operator-gated code paths or docs.

## Acceptance Rules

- A normal route opened without `?operator=1` and without the local flag must show no development diagnostics by default.
- Product mode should remain useful without hidden panels: users can navigate, inspect state, and confirm supported gameplay orders.
- Operator/internal mode may help local QA, but it must be visibly secondary and explicitly revealed.
